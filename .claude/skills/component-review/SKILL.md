## 역할

이 스킬은 React 컴포넌트 또는 훅을 코드 레벨에서 구조적으로 분석하는 파트너다.
분석 모드 선택 → 대상 파일 입력 → 분석 실행 순으로 진행한다.

---

## Step 1 — 분석 모드 선택

AskUserQuestion으로 다음 세 가지 중 하나를 선택받는다.

```
질문: "어떤 분석을 실행할까요?"

1. 관심사 분류
   이 컴포넌트가 담당하는 관심사를 UI / 상태 / 서버 통신 / 도메인 로직으로 분류

2. Custom Hook 분리 후보
   분리할 수 있는 Hook 후보를 찾고, 각 후보마다 "분리 이점"과 "분리하지 않아도 되는 이유"를 제시

3. 책임 과다 탐지
   Hook이 한 문장으로 설명되지 않는 책임을 갖고 있는지 검토하고 분리 후보 제안
```

---

## Step 2 — 분석 대상 입력

AskUserQuestion으로 분석할 파일을 입력받는다.

```
질문: "분석할 파일 경로나 컴포넌트명을 알려주세요."
      (예: src/market/CheckoutPage.tsx  또는  CheckoutPage)
```

- 전체 경로가 주어지면 바로 Read.
- 파일명만 주어지면 `find src -name "<입력값>.tsx" -o -name "<입력값>.ts"` 로 탐색 후 Read.
- 파일이 없으면 결과를 알리고 종료.

---

## Step 3 — 선택한 모드로 분석 실행

파일을 Read한 뒤 아래 해당 모드의 지침대로 분석한다.

---

### 모드 1: 관심사 분류

**목표**: 컴포넌트 코드에서 네 가지 관심사 영역을 식별하고, 혼재 위험 포인트를 짚는다.

**각 관심사 정의**

| 관심사      | 포함되는 것                                                      |
| ----------- | ---------------------------------------------------------------- |
| UI          | JSX 반환값, 조건부 렌더링, className/style, 이벤트 핸들러 바인딩 |
| 상태        | useState·useReducer 선언, 상태 업데이트 호출, 파생 상태 계산     |
| 서버 통신   | fetch·axios·react-query 호출, 로딩·에러 상태, 응답 파싱          |
| 도메인 로직 | 비즈니스 규칙, 데이터 변환·검증, 계산 로직 (렌더와 무관)         |

**출력 형식**

```
## 관심사 분류 — <파일명>

### UI
- [라인 X] <설명>

### 상태
- [라인 X-Y] <설명>

### 서버 통신
- 없음  /  [라인 X] <설명>

### 도메인 로직
- 없음  /  [라인 X] <설명>

### 혼재 위험 포인트
- [라인 X] <어떤 관심사가 뒤섞였는지 한 줄 설명>
  → 권장 처리: <분리 방향 한 줄>
```

혼재 포인트가 없으면 "없음" 으로 표기한다.

---

### 모드 2: Custom Hook 분리 후보

**목표**: 컴포넌트 안에서 Custom Hook으로 추출할 수 있는 코드 묶음을 찾고, 각 후보의 타당성을 균형 있게 평가한다.

**분리 후보 식별 기준**

- 상태(useState) + 부수 효과(useEffect)가 함께 묶인 로직
- 동일한 데이터 소스를 읽고 가공하는 코드 그룹
- 렌더링과 무관하게 독립적으로 동작하는 계산/검증 블록
- 다른 컴포넌트에서 재사용될 가능성이 있는 패턴

**평가 기준**

- **분리 이점**: 재사용성, 테스트 용이성, 관심사 단순화 중 실제로 해당하는 것만 기술
- **분리하지 않아도 되는 이유**: 코드 라인 수가 적음, 이 컴포넌트에서만 쓰임, 분리 시 오히려 props 전달이 복잡해짐 등 실제 근거 기술
- 근거 없는 "좋은 습관"류 주장은 쓰지 않는다

**출력 형식**

```
## Custom Hook 분리 후보 — <파일명>

### 후보 1: use<Name>
- **대상 코드**: 라인 X–Y
- **포함 내용**: <한 줄 요약>
- **분리 이점**:
  - <구체적 이점>
- **분리하지 않아도 되는 이유**:
  - <구체적 이유>
- **권장도**: 높음 / 중간 / 낮음
  이유: <한 줄>

### 후보 2: use<Name>
...

### 분리 후보 없음
<이유>
```

후보가 여러 개이면 권장도 높은 순으로 나열한다.

---

### 모드 3: 책임 과다 탐지

**목표**: Hook(또는 컴포넌트)이 단일 책임 원칙을 위반하는지 판단하고, 위반 시 분리 방향을 제안한다.

**판단 방법**

1. `"<훅/컴포넌트명>은 _____한다"` 형식으로 한 문장 설명을 시도한다.
   - 한 문장으로 완성되면 → 단일 책임 충족
   - "~하고, ~하며, ~도 처리한다" 식으로 늘어나면 → 책임 과다 의심

2. 코드에서 책임 단위를 직접 열거한다.

3. 각 책임에 대해 분리 후보 Hook을 제안한다.

**출력 형식**

```
## 책임 과다 탐지 — <파일명>

### 한 줄 설명 시도
"<이름>은 ___한다"
→ 결과: 단일 책임 충족 / 책임 과다 (X개 감지)

### 탐지된 책임 목록
1. <책임 A> — 라인 X–Y
2. <책임 B> — 라인 Z–W
...

### 분리 후보
| 책임 | 분리 Hook 이름 | 포함될 코드 |
|------|----------------|------------|
| <책임 A> | use<Name> | 라인 X–Y |
| <책임 B> | use<Name> | 라인 Z–W |

### 분리 우선순위
1. <가장 먼저 분리해야 할 것> — 이유: <한 줄>
2. ...

### 단일 책임 충족 시
"<이름>은 ___한다" — 분리 불필요.
```

---

## 공통 주의사항

- 분석은 실제 코드 라인을 근거로 한다. 추상적 설명만 쓰지 않는다.
- 이 프로젝트 컨벤션(함수형 컴포넌트, `handleXxx` 이벤트 핸들러, Suspense/ErrorBoundary 기반 비동기) 을 기준으로 판단한다.
- 분석 후 "추가로 다른 모드나 파일을 분석할까요?" 를 묻는다.
